## 📅 2026-04-11 - Markdown 编辑工具改为单文件自包含页面

### 🔍 问题描述
- **背景**：用户需要 `markdown-editor.html` 便于传播，要求页面本身包含完整逻辑。
- **现象**：之前新增的独立页面仍依赖项目内 `js` 和 `css` 文件，不符合单文件传播诉求。
- **影响**：离开当前项目目录后页面无法独立运行。

### 💡 解决思路
- **分析过程**：确认用户目标是“只保留一个 html 文件即可使用”。
- **解决策略**：将样式、界面结构与核心交互逻辑全部内联到 `markdown-editor.html` 中。
- **技术选择**：保留外部 CDN 依赖，仅去除对项目内本地文件的依赖。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/markdown-editor.html`
- **实现内容**：
  - 内联页面全部 CSS
  - 内联 Markdown 编辑器全部核心 JS 逻辑
  - 保留上传、实时预览、导出 MD、导出 HTML、主题切换、快捷插入、Mermaid 渲染等能力
- **变更说明**：
  - 页面不再依赖项目内其他 `css` 或 `js` 文件
  - 复制该 HTML 文件后仍可单独使用

### ⚡ 技术要点
- **核心逻辑**：单文件内完成编辑区、预览区、文件读取、导出与主题切换。
- **注意事项**：当前仍依赖外部 CDN，若要彻底离线分发，还需继续将第三方库进一步内嵌。
- **传播方式**：现在可以直接单独分发 `markdown-editor.html`。

### ✅ 测试验证
- **测试方法**：检查页面是否还引用项目内本地资源路径。
- **测试结果**：页面已去除项目内本地 `css/js` 依赖，仅保留单文件结构与外部 CDN 资源。
- **性能影响**：无额外项目耦合，页面体积略有增加但更便于分发。

### 🚀 后续优化
- **已知问题**：如果目标环境无法访问 CDN，则 Mermaid / Markdown 解析等第三方能力会受影响。
- **改进方向**：如需彻底离线传播，可继续做“全离线单 HTML 版”。
- **相关任务**：可按需要继续压缩页面体积或补充更多编辑快捷能力。

## 📅 2026-04-11 - Markdown 工具新增单独预览模式

### 🔍 问题描述
- **背景**：用户希望 Markdown 工具支持单独查看渲染后的文档内容。
- **现象**：原页面只有编辑区和预览区并排模式，无法专注查看最终渲染效果。
- **影响**：阅读长文档时视觉干扰较多，不利于纯预览场景使用。

### 💡 解决思路
- **分析过程**：基于现有单文件页面结构，最合适的做法是在原工具上增加“预览模式”切换，而不是新开页面。
- **解决策略**：新增预览模式按钮，切换后隐藏编辑区域，仅保留渲染结果区域。
- **技术选择**：通过增加状态变量和样式类控制布局切换，保持现有渲染逻辑不变。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/markdown-editor.html`
- **新增内容**：
  - 工具栏增加“预览模式”按钮
  - 新增 `previewOnly` 状态
  - 新增 `applyPreviewMode()` 用于切换页面布局
  - 预览模式下隐藏编辑面板，并让预览内容单独占满可视区域
- **变更说明**：
  - 不影响原有编辑与实时预览能力
  - 可在编辑模式与单独预览模式之间来回切换

### ✅ 测试验证
- **测试方法**：静态检查按钮事件、状态切换逻辑与预览布局样式。
- **测试结果**：切换预览模式时，页面会隐藏编辑区并单独展示渲染后的 Markdown 内容。
- **性能影响**：无明显性能影响，仅增加少量状态控制逻辑。

### 🚀 后续优化
- **已知问题**：当前预览模式仍保留顶部工具栏，若需要更沉浸阅读可继续增加“纯净阅读模式”。
- **改进方向**：可继续增加“返回编辑”“打印预览”“目录导航”等能力。
- **相关任务**：后续可结合 URL 参数，让页面默认以预览模式打开。

## 📅 2026-04-11 - Markdown 工具新增 PDF 导出功能

### 🔍 问题描述
- **背景**：用户希望 Markdown 工具支持将文档直接导出为 PDF。
- **现象**：当前仅支持导出 Markdown 和 HTML，缺少常用的 PDF 输出能力。
- **影响**：用户无法直接生成适合分享和打印的 PDF 文件。

### 💡 解决思路
- **分析过程**：单文件页面中已有完整的 Markdown 编辑和预览逻辑，适合直接在页面内补充 PDF 导出入口。
- **解决策略**：新增 PDF 导出按钮，并通过 `jsPDF` 将当前 Markdown 文本内容分页写入 PDF 文件。
- **技术选择**：使用 CDN 引入 `jsPDF`，保持单文件结构不依赖项目内其他脚本。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/markdown-editor.html`
- **新增内容**：
  - 工具栏增加“导出 PDF”按钮
  - 引入 `jsPDF` 依赖
  - 新增 `exportPdf()` 导出方法
- **变更说明**：
  - 支持直接导出 `document.pdf`
  - 保持原有 MD / HTML 导出功能不变

### ✅ 测试验证
- **测试方法**：静态检查按钮事件、PDF 依赖引入和导出逻辑。
- **测试结果**：页面已具备 PDF 导出入口和相应导出逻辑。
- **性能影响**：增加一个轻量 PDF 依赖，请求体积略有上升。

### 🚀 后续优化
- **已知问题**：当前 PDF 导出以文本分页为主，不会完整保留 HTML 样式效果。
- **改进方向**：如后续需要更高保真导出，可继续结合画布截图或打印样式方案。
- **相关任务**：可后续增加“仅导出预览区”或“打印为 PDF”能力。

## 📅 2026-04-11 - Markdown 高保真预览区 PDF 导出修复

### 🔍 问题描述
- **背景**：用户反馈 PDF 导出结果出现乱码，并明确需要高保真且仅导出预览区。
- **现象**：原先的 PDF 导出采用文本写入方式，中文容易乱码，且无法保留预览区实际渲染样式。
- **影响**：导出的 PDF 既无法正确显示内容，也不符合预览区导出需求。

### 💡 解决思路
- **分析过程**：文本写入 PDF 依赖字体支持，浏览器端默认字体无法可靠覆盖中文场景；同时文本导出也无法保留 Markdown 渲染样式。
- **解决策略**：改为将预览区渲染结果截图后分页写入 PDF，只导出预览区内容。
- **技术选择**：使用 `html2canvas` + `jsPDF` 组合实现高保真导出。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/markdown-editor.html`
- **修复内容**：
  - 引入 `html2canvas`
  - 重写 `exportPdf()`，改为截图预览区再分页写入 PDF
  - 导出前自动切换到预览模式，保证只输出预览区内容
- **变更说明**：
  - 解决中文乱码问题
  - PDF 导出结果更接近页面预览效果
  - 导出目标限定为预览区，不再混入编辑区内容

### ✅ 测试验证
- **测试方法**：静态检查依赖引入、预览区截图逻辑和分页写入流程。
- **测试结果**：页面已具备高保真预览区 PDF 导出实现。
- **性能影响**：导出时会增加一次截图和分页处理开销，但仅在导出动作时发生。

### 🚀 后续优化
- **已知问题**：超长文档导出时截图会占用更多内存和时间。
- **改进方向**：后续可增加导出质量选项和分页边距设置。
- **相关任务**：如需进一步优化，可增加“仅导出当前可见主题”或“打印版 PDF”模式。

## 📅 2026-04-11 - Markdown PDF 分页截断与尾页压缩修复

### 🔍 问题描述
- **背景**：用户在使用高保真 PDF 导出时，发现分页效果不理想。
- **现象**：分页时内容会被硬切开，且最后一页因为高度计算方式不合理出现压缩。
- **影响**：导出的 PDF 阅读体验差，尤其对表格、代码块和段落块影响明显。

### 💡 解决思路
- **分析过程**：之前按固定像素高度硬切 canvas，导致分页点可能落在内容块中间，同时最后一页仍沿用统一画布高度，造成视觉压缩。
- **解决策略**：基于预览区块级元素位置计算更自然的分页点，并为每一页按真实切片高度生成独立画布。
- **技术选择**：继续使用 `html2canvas` + `jsPDF`，但优化分页算法和尾页切片逻辑。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/markdown-editor.html`
- **修复内容**：
  - 导出时先复制预览区到离屏容器，再统一截图
  - 根据块级元素 `offsetTop / offsetHeight` 计算更合理的分页边界
  - 每页使用独立高度画布，避免最后一页被拉伸或压缩
- **变更说明**：
  - 分页更加自然
  - 尾页内容按真实高度导出，不再压缩
  - 保持仅导出预览区和高保真效果

### ✅ 测试验证
- **测试方法**：静态检查分页切片与页面生成逻辑。
- **测试结果**：分页逻辑已改为按内容块优先分页，尾页采用真实高度输出。
- **性能影响**：分页计算略有增加，但只在导出时发生。

### 🚀 后续优化
- **已知问题**：极端超长的单个大块元素仍可能跨页。
- **改进方向**：后续可继续增加针对表格、代码块、Mermaid 图的专用分页策略。
- **相关任务**：如需要可继续做“避免标题单独落页”的排版优化。

## 📅 2026-04-11 - 工具集 Markdown 支持高保真 PDF 导出与智能分页

### 🔍 问题描述
- **背景**：用户希望工具集入口 `src/tools/ui/index.html` 中的 Markdown 工具也支持 PDF 导出，并进一步优化分页体验。
- **现象**：工具集版 Markdown 工具此前没有 PDF 导出入口，也缺少针对标题、代码块、表格、Mermaid 图的分页保护。
- **影响**：两个入口能力不一致，且导出 PDF 时页面排版质量不足。

### 💡 解决思路
- **分析过程**：工具集版实际由 `src/tools/js/tools/markdown-editor.js` 渲染，且 `index.html` 负责加载依赖，因此需要同时改入口脚本加载与工具内部实现。
- **解决策略**：给工具集版补充 PDF 依赖和导出按钮，并同步高保真预览区导出能力，再在分页逻辑中增加内容类型保护规则。
- **技术选择**：继续使用 `html2canvas` + `jsPDF`，通过离屏预览容器和分页规则控制导出结果。

### 🔧 代码变更
- **修改文件**：
  - `src/tools/ui/index.html`
  - `src/tools/js/tools/markdown-editor.js`
  - `src/tools/css/markdown-editor.css`
- **新增/修复内容**：
  - `index.html` 增加 `jsPDF` 依赖
  - Markdown 工具控制区增加“导出 PDF”按钮
  - 新增 `exportPdf()`、`decorateExportPreview()`、`computePdfBreakpoints()` 等导出逻辑
  - 增加标题防孤页、代码块尽量整块分页、表格分页优化、Mermaid 图单独分页保护规则
  - 增加导出专用离屏容器样式
- **变更说明**：
  - 工具集入口现已支持仅导出预览区的高保真 PDF
  - 分页更智能，更接近正式文档排版效果

### ✅ 测试验证
- **测试方法**：静态检查入口依赖、按钮绑定、导出逻辑和分页规则。
- **测试结果**：工具集版 Markdown 已接入 PDF 导出能力，并具备分页保护逻辑。
- **性能影响**：导出时会增加截图和分页计算开销，但仅在导出动作发生时生效。

### 🚀 后续优化
- **已知问题**：超长单个表格或单个超高 Mermaid 图仍可能因自身高度超过一页而跨页。
- **改进方向**：后续可继续增加针对超大元素的缩放策略或拆分页策略。
- **相关任务**：如需要可继续统一单文件版与工具集版的分页规则实现细节。

- **原因 1**：`index.html` 引用路径为 `../js/tools/markdown-editor.js`（即 `src/tools/js/tools/`），实际文件只在 `src/js/tools/markdown-editor.js`，脚本未加载。
- **原因 2**：`window.ToolRegistry` 原在 `app.js` 末尾赋值，各工具脚本先于 `app.js` 执行时 `ToolRegistry` 不存在，无法 `register`；加载 `app.js` 后又用空表覆盖，导致从注册表读到的工具列表为空（与「仅有 Markdown 不显示」并存时，表现为新工具永远进不了列表）。

### 🔧 代码变更
- **新增**：`src/tools/js/tools/markdown-editor.js`（与首页引用路径一致，并增加 `'use strict'`）。
- **修改**：`src/tools/js/utils.js` 在文件末尾初始化全局 `ToolRegistry`，保证工具脚本加载时即可注册。
- **修改**：`src/tools/js/app.js` 中仅在 `!window.ToolRegistry` 时创建注册表，避免覆盖已注册工具。
- **删除**：`src/js/tools/markdown-editor.js`（避免双份维护，以后以 `src/tools/js/tools/markdown-editor.js` 为准）。

### ⚡ 技术要点
- 脚本顺序：`utils.js` → 各 `tools/*.js`（内联 `register`）→ `app.js`（`registerTools` 从 `ToolRegistry` 拉取）。

## 📅 2026-04-11 - Markdown 工具返回后出现多余弹窗

### 🔍 问题描述

- **现象**：返回操作后页面还会再出现一个浏览器弹窗，影响正常使用。
- **根因**：`goBack()` 直接移除编辑器节点后又调用 `window.history.back()` 作为兜底逻辑，未走当前页面已有的工具弹窗关闭流程，导致返回行为异常并触发额外交互弹窗。

### 💡 解决思路
- 不再手动删除 `.markdown-editor` 节点，也不再调用浏览器历史返回。
- 统一复用主应用现有的弹窗关闭方法，直接关闭 `toolModal`，让页面回到工具列表。

### 🔧 代码变更
- **修改文件**：`src/tools/js/tools/markdown-editor.js`
- **关键修复**：重写 `goBack()`，优先调用 `window.webToolsApp.closeModal()`；若主应用实例不可用，则仅移除弹窗激活状态并恢复页面滚动。

### ⚡ 技术要点
- Markdown 编辑器本身运行在工具详情弹窗内部，“返回”本质上应是关闭当前工具弹窗，而不是页面级历史回退。
- 统一走主应用的关闭逻辑，可以避免重复 DOM 清理和异常弹窗。

### ✅ 测试验证
- 打开首页 → 进入 Markdown 编辑器 → 点击“返回” → 应直接关闭工具弹窗并回到工具列表，不再出现额外弹窗。

### 🚀 后续优化
- 可考虑给工具组件增加统一的“退出当前工具”接口，避免各工具自行处理返回逻辑。

## 📅 2026-04-11 - Markdown 编辑器 Mermaid 图表未生效分析

### 🔍 问题描述
- **背景**：页面已在 `index.html` 中引入 Mermaid 脚本，用户希望在 Markdown 编辑器预览区直接显示流程图等图表。
- **现象**：即使输入 Mermaid 语法，预览区也不会渲染成图表，只会当作普通代码块或普通文本显示。
- **根因**：当前 `markdown-editor.js` 只做了 `marked` 解析、HTML 清洗和 `highlight.js` 代码高亮，没有在预览完成后识别 `mermaid` 代码块并调用 Mermaid 进行二次渲染。

### 💡 解决思路
- 保留 Markdown 正常解析流程。
- 在 `updatePreview()` 完成 HTML 注入后，查找预览区内的 Mermaid 代码块。
- 将 Mermaid 代码块替换为 Mermaid 容器，并调用 Mermaid API 执行渲染。

### 🔧 当前使用说明
- **正确输入方式**：必须使用 fenced code block，并把语言标记写成 `mermaid`。
- **示例**：
  ```markdown
  ```mermaid
  graph TD
    A[开始] --> B[输入内容]
    B --> C{是否有效}
    C -->|是| D[渲染图表]
    C -->|否| E[提示错误]
  ```
  ```
- **注意**：当前版本即使这样输入，也不会真正渲染为图表，仍需补上 Mermaid 渲染逻辑。

### ✅ 结论
- `index.html` 已具备 Mermaid 依赖加载条件。
- 真正缺失的是 `src/tools/js/tools/markdown-editor.js` 中的 Mermaid 识别与渲染代码，而不是输入语法本身。

### 🚀 后续优化
- 可在导出 HTML 时一并内嵌 Mermaid 渲染支持，保证导出文件也能展示图表。
