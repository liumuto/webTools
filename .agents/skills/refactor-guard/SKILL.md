# Refactor Guard

用于在开发过程中约束重构冲动，防止“为了解决一个点，顺手改一大片”。

## 使用时机

- 修复 bug 时
- 为小功能增加少量逻辑时
- 面对长文件但用户未要求系统重构时

## 判定标准

只有满足以下至少一项，才允许重构：
- 当前需求无法在现结构下安全实现
- 重复逻辑已经直接阻碍本次修改
- 不重构会明显增加本次 bug 风险

## 重构要求

- 范围必须与当前需求直接相关。
- 每次重构都要说明必要性。
- 优先局部提取，不做全局搬迁。
- 保持外部行为不变。

## 应避免的行为

- 仅因“看起来不顺眼”就改结构。
- 把单次需求升级成抽象工程。
- 新建只被调用一次的工具层或助手层。

## 输出要求

如果进行了重构，需补充：
- 为什么必须重构
- 重构范围
- 如何确认外部行为未变
