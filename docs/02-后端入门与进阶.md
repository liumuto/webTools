### 2.3 后端必懂的几种结构

**(1) 哈希表 - 万能加速器**

```javascript
// Node.js: 用 Map 缓存用户信息
const userCache = new Map();                 // 内置 Map
function getUser(id) {
  if (userCache.has(id)) return userCache.get(id);  // 命中缓存
  const u = db.queryUser(id);                       // 回源数据库
  userCache.set(id, u);                             // 写入缓存
  return u;
}
```

**(2) 跳表 - Redis 有序集合的底层**

```
Level 3:  1 ------------------> 9
Level 2:  1 ------> 4 --------> 9
Level 1:  1 -> 3 -> 4 -> 6 ---> 9
Level 0:  1 -> 3 -> 4 -> 6 -> 8 -> 9
```

跳表在期望 O(log n) 下完成有序操作, 实现比红黑树简单, 是 Redis ZSet 的核心。

**(3) B+ 树 - MySQL InnoDB 索引基石**

```mermaid
flowchart TD
  Root["根: [10, 20]"] --> L1["[1, 5, 10]"]
  Root --> L2["[10, 15, 20]"]
  Root --> L3["[20, 25, 30]"]
  L1 -.叶子链表.-> L2
  L2 -.叶子链表.-> L3
```

B+ 树特点:
- 非叶子节点只存索引, 叶子节点存数据
- 叶子节点形成有序链表, 范围查询超快
- 一般 3-4 层即可索引千万行数据

**(4) 布隆过滤器 - 防缓存穿透**

```python

    return uid in bf                         # True = 可能存在, False = 一定不存在
```

**Leetcode 1 · 两数之和 - 哈希一遍过**

```go
// Go 实现, 时间复杂度 O(n)
func twoSum(nums []int, target int) []int {
    m := make(map[int]int)                   // value -> index
    for i, n := range nums {
        if j, ok := m[target-n]; ok {        // 找到配对
            return []int{j, i}
        }
        m[n] = i
    }
    return nil
}
```

**Leetcode 146 · LRU 缓存 (高频)**

```java
// Java: 哈希 + 双向链表, 经典手撕题
class LRUCache {
    class Node { int k, v; Node prev, next; }
    private final int cap;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(), tail = new Node();

    public LRUCache(int capacity) {
        this.cap = capacity;
        head.next = tail; tail.prev = head;   // 哨兵节点
    }
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node n = map.get(key);
        moveToHead(n);
        return n.v;
    }
    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node n = map.get(key);
            n.v = value;
            moveToHead(n);
            return;
        }
        Node n = new Node();
        n.k = key; n.v = value;
        addToHead(n);
        map.put(key, n);
        if (map.size() > cap) {
            Node last = tail.prev;
            removeNode(last);
            map.remove(last.k);
        }
    }
    private void addToHead(Node n) {
        n.prev = head; n.next = head.next;
        head.next.prev = n; head.next = n;
    }
    private void removeNode(Node n) {
        n.prev.next = n.next; n.next.prev = n.prev;
    }
    private void moveToHead(Node n) { removeNode(n); addToHead(n); }
}
```

### 3.2 TCP 三次握手与四次挥手

```mermaid
sequenceDiagram
  participant C as Client
  participant S as Server
  Note over C,S: 三次握手 - 建立连接
  C->>S: SYN, seq=x
  S->>C: SYN+ACK, seq=y, ack=x+1
  C->>S: ACK, ack=y+1
  Note over C,S: 数据传输
  C->>S: 数据 + ACK
  S->>C: 数据 + ACK
  Note over C,S: 四次挥手 - 关闭连接
  C->>S: FIN, seq=u
  S->>C: ACK, ack=u+1
  S->>C: FIN, seq=v
  C->>S: ACK, ack=v+1
```

⚠️ **踩坑**:
- 为什么三次握手? 防止历史连接重复初始化, 同步序号
- 为什么四次挥手? 服务端可能还有数据要发, 需分两次关闭
- **TIME_WAIT** 状态持续 2MSL, 高并发场景需调内核参数

## 第 4 章 操作系统基础

### 4.1 进程 vs 线程 vs 协程

```mermaid
flowchart TD
  P[进程<br/>独立内存空间] --> T1[线程 1]
  P --> T2[线程 2]
  P --> T3[线程 3]
  T1 --> G1[协程 a]
  T1 --> G2[协程 b]
```

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 切换成本 | 大 (页表/上下文) | 中 (寄存器/栈) | 极小 (用户态) |
| 资源隔离 | 强 | 弱(共享进程内存) | 弱(共享线程栈或独立小栈) |
| 调度 | 内核 | 内核 | 用户态调度器 |
| 数量级 | 几千 | 几万 | 几十万 - 百万 |
| 通信 | IPC (管道/socket/共享内存) | 共享变量+锁 | channel / future |

### 4.2 I/O 模型对比 (网络编程基石)

```mermaid
flowchart LR
  subgraph BIO[阻塞 I/O]
    A1[read 调用] -->|阻塞等待| A2[数据返回]
  end
  subgraph NIO[非阻塞 I/O]
    B1[read 调用] -->|立即返回 EAGAIN| B2[轮询]
  end
  subgraph IO_MUX[I/O 多路复用]
    C1[select/poll/epoll] -->|事件通知| C2[处理就绪的 fd]
  end
  subgraph AIO[异步 I/O]
    D1[发起 read] -->|完成回调| D2[数据已就绪]
  end
```

| 模型 | 同步/异步 | 阻塞/非阻塞 | 代表 |
|------|----------|-------------|------|
| BIO | 同步 | 阻塞 | Java Socket |
| NIO | 同步 | 非阻塞 | Java NIO, epoll level-triggered |
| IO 多路复用 | 同步 | 非阻塞 | epoll, kqueue |
| AIO | 异步 | 非阻塞 | io_uring, Windows IOCP |

### 4.3 select / poll / epoll 对比

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| fd 数量上限 | 1024 (FD_SETSIZE) | 无上限 | 无上限 |
| 时间复杂度 | O(n) 轮询 | O(n) 轮询 | O(1) 就绪通知 |
| 内核态拷贝 | 每次都拷贝 fd 集 | 同左 | 仅注册一次 |
| 触发模式 | LT | LT | LT + ET |
| 适用场景 | 跨平台兼容 | 兼容 | 高并发 (Linux) |

```c
// epoll 三件套伪代码
int epfd = epoll_create1(0);
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);     // 注册感兴趣的 fd
int n = epoll_wait(epfd, events, MAX, -1);   // 等待事件
for (int i = 0; i < n; i++) handle(events[i]);
```

📌 **重点**: Nginx、Redis、Netty 高性能的核心都是 epoll + 事件驱动。

### 4.4 内存管理

```
+------------------+ 高地址
|     内核空间      |
+------------------+
|       栈         | <- 函数局部变量, 向下增长
|        ↓         |
|                  |
|        ↑         |
|       堆         | <- malloc/new, 向上增长
+------------------+
|    BSS 段        | <- 未初始化全局
|    Data 段       | <- 已初始化全局
|    Text 代码段   |
+------------------+ 低地址
```

🎯 **后端关注的内存问题**:
- 内存泄漏: Node 闭包持有大对象、Java 静态 Map 越长越大
- 内存溢出: JVM OOM 时看 dump
- 内存碎片: Redis 长期运行碎片率 (mem_fragmentation_ratio)

### 4.5 进程间通信 IPC

| 方式 | 速度 | 跨主机 | 典型用途 |
|------|------|--------|----------|
| 管道 Pipe | 快 | 否 | Shell 命令 |
| 命名管道 FIFO | 快 | 否 | 同机进程通信 |
| 消息队列 | 中 | 否 | POSIX MQ |
| 共享内存 + 信号量 | 极快 | 否 | 高性能 IPC |
| Unix Socket | 中 | 否 | Docker / Nginx FCGI |
| TCP/UDP Socket | 中 | 是 | 分布式 |

- 后端性能 = I/O 模型 + 多路复用 + 线程/协程调度

### 5.4 索引原理 (B+ 树)

```mermaid
flowchart TD
  R["根节点<br/>[40, 70]"]
  R --> N1["[10, 20, 30]"]
  R --> N2["[40, 50, 60]"]
  R --> N3["[70, 80, 90]"]
  N1 --> L1["叶子: 10 数据"]
  N1 --> L2["叶子: 20 数据"]
  N1 --> L3["叶子: 30 数据"]
  L1 -.next.-> L2 -.next.-> L3
```

- **聚簇索引 (Clustered Index)**: 叶子节点存完整行数据, 一张表只能有一个 (主键)
- **二级索引 (Secondary Index)**: 叶子节点存主键, 回表两次

⚠️ **踩坑**:
- 频繁更新的字段不要建索引
- 联合索引遵循"最左前缀"原则
- `LIKE '%abc'` 索引失效, `LIKE 'abc%'` 可用
- `OR` 条件如果有一边没索引, 全索引失效
- 类型隐式转换会让索引失效 (`WHERE phone = 13800000000` 而字段是 VARCHAR)

### 5.5 SQL 优化套路

📌 **EXPLAIN 关键字段**:

| key | 实际使用的索引 | 非 NULL |

```java
// Java 缓存击穿解决: 互斥锁
public Product getProduct(Long id) {
    String key = "product:" + id;
    Product p = redis.get(key);
    if (p != null) return p;

    String lockKey = "lock:" + key;
    if (redis.setnx(lockKey, "1", 10, TimeUnit.SECONDS)) {
        try {
            p = db.findById(id);
            redis.set(key, p, 60, TimeUnit.SECONDS);
            return p;
        } finally {
            redis.del(lockKey);
        }
    } else {
        Thread.sleep(50);
        return getProduct(id);                // 重试
    }
}
```

### 9.2 Goroutine + Channel

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

// 生产者-消费者
func main() {
    ch := make(chan int, 10)                 // 带缓冲 channel
    var wg sync.WaitGroup

    // 启动 3 个消费者
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for v := range ch {              // channel 关闭后退出
                fmt.Printf("consumer %d got %d\n", id, v)
                time.Sleep(100 * time.Millisecond)
            }
        }(i)
    }

    // 生产
    for i := 0; i < 20; i++ {
        ch <- i
    }
    close(ch)                                // 关闭 channel
    wg.Wait()
}
```

⚠️ **踩坑**:
- 关闭 channel 是生产者的责任, 不要在消费端关
- 向已关闭 channel 发送会 panic
- nil channel 读写永久阻塞 (可用于 select 屏蔽分支)

## 第 11 章 认证与授权

### 11.1 概念区分

- **认证 Authentication**: 你是谁?
- **授权 Authorization**: 你能做什么?
- **会话 Session**: 你在多长时间内保持已登录

### 11.2 主流方案对比

| 方案 | 状态 | 适用 |
|------|------|------|
| Session + Cookie | 有状态 | 单体应用, 小流量 |
| JWT | 无状态 | 分布式, 跨域 |
| OAuth2 | 协议 | 第三方登录, 开放平台 |
| OIDC | 协议 (基于 OAuth2) | SSO, 联邦登录 |
| SAML | 协议 (XML) | 企业 SSO |

### 11.3 JWT 结构

```
header.payload.signature

header   eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
payload  eyJzdWIiOiIxIiwibmFtZSI6ImFsaWNlIiwiZXhwIjoxNzMwMDAwMDAwfQ
sig      <HMAC-SHA256(header + "." + payload, secret)>
```

```javascript
// Node 签发 JWT
const jwt = require('jsonwebtoken');
const token = jwt.sign(
  { sub: 1, name: 'alice', role: 'admin' },
  process.env.JWT_SECRET,
  { expiresIn: '2h', issuer: 'my-app' }
);

// 验证
try {
  const payload = jwt.verify(token, process.env.JWT_SECRET);
  console.log(payload);
} catch (e) {
  console.error('invalid token', e.message);
}
```

⚠️ **JWT 踩坑**:
- 不要把敏感信息放 payload (Base64 可解码, 非加密)
- 想"主动退出"必须配合服务端黑名单
- 使用 HTTPS, 防中间人嗅探
- 不要使用 `alg: none`, 不要把签名算法写在 header 让攻击者控制

### 11.4 OAuth2 授权码流程

```mermaid
sequenceDiagram
  participant U as User
  participant C as Client (App)
  participant AS as Auth Server
  participant RS as Resource Server
  U->>C: 点击"用 Google 登录"
  C->>AS: 重定向, response_type=code & client_id & redirect_uri & scope & state
  AS->>U: 显示授权页
  U->>AS: 同意授权
  AS->>C: 重定向回 redirect_uri?code=xxx&state=xxx
```

OAuth2 四种授权模式:

3. **密码模式 (ROPC)** - 内部信任应用
4. **客户端模式 (Client Credentials)** - 服务间调用

## 第 13 章 分布式基础

### 13.1 CAP 定理

```mermaid
flowchart TD

  A --- P[Partition Tolerance 分区容错]
  P --- C
```

| 偏向 | 代表 | 场景 |
|------|------|------|
| CP | ZooKeeper, Etcd, HBase | 强一致, 容忍不可用 |
| AP | Eureka, Cassandra, DynamoDB | 高可用, 容忍最终一致 |

### 13.2 BASE 理论

- **B**asically Available 基本可用
- **S**oft state 软状态
- **E**ventually consistent 最终一致

### 13.3 一致性算法

**Raft 简介**:

```mermaid
stateDiagram-v2
  [*] --> Follower
  Follower --> Candidate: 选举超时
  Candidate --> Leader: 获多数票
  Candidate --> Follower: 发现更高 term
  Leader --> Follower: 发现更高 term
```

Raft 核心:
1. **Leader 选举**: 任期 term 递增, 随机超时
2. **日志复制**: Leader 把命令复制到多数节点才 commit
3. **安全性**: 已 commit 的日志不会被覆盖

经典实现: etcd, Consul, TiKV, RocketMQ 5.0。

**Paxos** 是 Raft 的"祖先", 更通用但更难理解。

### 13.4 分布式 ID

| 方案 | 优点 | 缺点 |
|------|------|------|
| UUID | 简单, 无中心 | 无序, 占空间 36 字节 |
| 数据库自增 | 简单 | 单点瓶颈 |
| Redis INCR | 高性能 | 需高可用 |
| 雪花 Snowflake | 趋势递增, 64bit | 时钟回拨需处理 |
| 美团 Leaf | 双 buffer 段 | 部署 |
| 百度 UidGenerator | RingBuffer | 复杂 |

Snowflake 64 位:

```
| 1 bit unused | 41 bit timestamp | 10 bit machine | 12 bit seq |
```

```go
// Go 简化版 Snowflake
type Snowflake struct {
    mu        sync.Mutex
    timestamp int64
    machineId int64
    sequence  int64
}

const (
    machineBits = 10
    seqBits     = 12
    seqMask     = -1 ^ (-1 << seqBits)
    twepoch     = int64(1577836800000) // 2020-01-01
)

func (s *Snowflake) NextID() int64 {
    s.mu.Lock()
    defer s.mu.Unlock()
    now := time.Now().UnixMilli()
    if now == s.timestamp {
        s.sequence = (s.sequence + 1) & seqMask
        if s.sequence == 0 {
            for now <= s.timestamp { now = time.Now().UnixMilli() }
        }
    } else {
        s.sequence = 0
    }
    s.timestamp = now
    return ((now - twepoch) << (machineBits + seqBits)) |
        (s.machineId << seqBits) | s.sequence
}
```

### 16.3 经典案例: 订单聚合

```java
// Java 订单聚合根
public class Order { // AggregateRoot
    private OrderId id;                       // 实体标识
    private CustomerId customerId;
    private List<OrderItem> items;            // 内部实体
    private Money totalAmount;                // 值对象
    private OrderStatus status;

    // 保持不变式
    public void addItem(ProductId productId, int quantity, Money price) {
        if (status != OrderStatus.DRAFT) throw new BizException("已下单不能修改");
        items.add(new OrderItem(productId, quantity, price));
        recalculate();
    }

    public void pay() {
        if (status != OrderStatus.PENDING) throw new BizException("状态不对");
        this.status = OrderStatus.PAID;
        DomainEventBus.publish(new OrderPaidEvent(this.id));
    }

    private void recalculate() {
        this.totalAmount = items.stream()
            .map(OrderItem::subtotal)
            .reduce(Money.ZERO, Money::add);
    }
}
```

📌 后端是一个"广而深"的方向:

1. **不要怕底层**: 越底层越通用, 越值钱
2. **不要拒变化**: 工具会过时, 原理常青
3. **不要孤军奋战**: 多写、多分享、多看源码、多参与开源
